# Промпт для репозитория wenay-react2: темизация мышиного контекст-меню (MenuBase)

Скопируй текст ниже в сессию в репе wenay-react2.

---

Нужно перевести мышиное контекст-меню (правый клик: `mouseMenuApi.ReactMouse` →
`MenuR` → `MenuBase`/`MenuElement`, файлы `src/common/src/menu/menu.tsx` /
`menuR.tsx`) на темизируемый токен-контракт `--menu-*`. Сейчас при смене темы
приложения меню остаётся в легаси-виде и выбивается из макета.

## Контекст

Приложения-потребители переключают темы атрибутом `[data-theme]` на `<html>` и
перекрывают токены библиотеки селектором `:root[data-theme='...']`. Контракт
`--menu-*` уже существует и уже темизируется приложениями — его использует
ПРАВОЕ меню ☰ (`menuRight.css`: `.dropdown-content` / `.menu-item`):

- `--menu-bg-color` — фон меню;
- `--menu-border` — рамки;
- `--menu-item-color` — цвет пункта;
- `--menu-item-hover-color` / `--menu-item-hover-bg-color` — hover пункта;
- `--menu-active-background` — активное состояние;
- `--menu-shadow` — тень.

А мышиное меню (MenuBase) стилизуется легаси-классами из `style.css` и токены
использует только частично:

- контейнер `.MenuR` — берёт `--menu-bg-color` и `--menu-shadow` (ок);
- пункты — классы `.toButton` / `.toButtonA`: цвет текста наследуется (не
  `--menu-item-color`), hover = подчёркивание `border-bottom` + цвет
  `--color-text-theme` (а не инверсия `--menu-item-hover-*`, как у `.menu-item`);
- `.toButtonA:hover { color: #ffffff }` — захардкоженный белый, в светлой теме
  пункт «исчезает»;
- рамка пунктов не использует `--menu-border`.

## Задача

1. Перевести стили пунктов MenuBase на тот же контракт `--menu-*`, что и
   `.menu-item` правого меню:
   - пункт: цвет `--menu-item-color`, разделитель `border-bottom: var(--menu-border)`;
   - hover/активный пункт: фон `--menu-item-hover-bg-color`, цвет
     `--menu-item-hover-color` (инверсия, как в `.menu-item:hover`);
   - активное состояние (`toButtonA` / `active?.()`): `--menu-active-background`;
   - убрать все захардкоженные цвета (`#ffffff` и т.п.) — только `var(...)`
     с фолбэком на текущее значение.
2. Дефолты токенов НЕ менять: без переопределений тем меню должно выглядеть
   как сейчас (приложения без тем не должны измениться).
3. Совместимость API: имена классов (`MenuR`, `toButton`, `toButtonA`) и пропсы
   (`className`, `menuElement` у MenuBase/ReactMouse) не переименовывать —
   на них завязаны приложения. Меняем только сами CSS-правила.
4. Если появятся новые токены — объявить дефолты в инжекте токенов библиотеки
   (`tokens.css`) + зеркало в `src/common/src/styles/tokens.ts`, по образцу
   существующих `--menu-*`.

## Опционально (отдельным коммитом, если по силам)

`MenuR` ловит правый клик только внутри собственного поддерева (React
`onMouseUp` на обёртке `div.maxSize`). Контент, отрендеренный вне обёртки
(стеки плавающих окон, порталы), открыть меню не может — приложениям приходится
затаскивать хосты окон внутрь `ReactMouse`. Хочется опцию:

- либо проп `captureGlobal?: boolean` — слушатели `mouseup`/`contextmenu` на
  `document` (с тем же гейтом `statusOn` и подавлением нативного меню),
- либо экспортируемый императивный API `mouseMenuApi.open({x, y})` — открыть
  меню с текущим содержимым `map` из любого места.

Поведение по умолчанию не менять.
