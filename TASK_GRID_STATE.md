# Задача: слой состояния грида + мобильный селектор колонок

> Файл задачи и журнал прогресса. Обновлять после каждого значимого шага —
> предыдущие попытки терялись из-за перезагрузок компьютера.

## Постановка (со слов заказчика)

1. **Сохранение состояния ag-grid отдельным слоем.** Пользователь двигает
   колонки, меняет ширину, скрывает и т.п. — состояние должно сохраняться и
   восстанавливаться. Слой НЕ встраивается в `agGrid4` (обёртка не трогается),
   а подключается снаружи к живому гриду (через gridApi/события).
2. **Мобильный режим.** На телефоне таблица со 100–120 строками и многими
   колонками нечитаема, горизонтальный скролл неудобен. Нужен селектор
   видимых колонок:
   - вариант «ползунок с точками»: точки на треке = выбранные колонки
     (2–4 точки), где точка стоит — тот столбец и показывается;
   - добавление/удаление точек — изящным жестом (свайп точки вверх,
     drag-off и т.п.), НЕ кнопками «+/−»;
   - выбранные колонки показываются не таблицей, а блоками/карточками.
3. **Вариант с иконками.** Альтернативная/дополнительная реализация: меню
   иконок (у нас уже есть хук перестановки иконок `useReorder`), где иконки =
   колонки. Требуется **двусторонняя связь** с живым гридом: меню показывает,
   какие колонки реально видимы/скрыты, а клик по иконке скрывает/показывает
   колонку в гриде. Открыл настройки → вычеркнул ненужные, оставил нужные.
2а. **Уточнение по мобильному режиму (2026-07-07, со слов заказчика).**
   Мобильный вид — не обязательно таблица (карточки/блоки), но принцип тот же:
   ползунком настраиваются видимые поля. Дополнительно — **сортировка**:
   - tap по точке (без перемещения) = **выделить** это поле;
   - кнопка «Сортировка» циклично переключает сортировку по выделенному полю
     (по возр. → по убыв. → выкл., «чик-чик-чик»);
   - сортировка **липкая**: включил по полю X → выделил потом другую точку —
     сортировка остаётся по X, пока явно не включена новая. Т.е. сортировка
     не зависит от текущего выделения и от набора видимых полей (сортировать
     можно и по скрытому полю).

3а. **Уточнения по меню иконок (2026-07-07, со слов заказчика).**
   - **Зеркало 1:1**: переставил колонки в гриде → кнопки в меню переставились
     так же (и наоборот). Отражение порядка двустороннее.
   - **Слои, не монстр**: меню строится из нескольких слоёв. Нижний слой
     (полоса кнопок) НЕ знает, что означает клик по кнопке — «это история
     отдельного слоя». Он только рисует состояния и репортит клики/drag.
   - **Кнопка живёт дольше колонки**: если колонку удалили из грида (например,
     стандартом «убрать пустые»), кнопка остаётся на месте, но НЕАКТИВНА.
   - **Состояния кнопки**: неактивная / выключенная / включённая / включённая
     «с прибамбасами» (адорнмент: поле с подполями, стратегии наименований
     и т.п. — точки/маркеры у кнопки).
   - **Стандарты таблицы** (пример многосостоянной кнопки): у таблицы есть
     режимы отображения, обычно три: (1) показывать всё; (2) убирать пустые
     столбцы, 0 считается пустым; (3) убирать пустые столбцы, 0 — значение.
     Кнопка циклично переключает режим («чик-чик-чик»).

3б. **Уточнение: Toolbar поверх columnState (2026-07-07, со слов заказчика).**
   - У `createToolbar` (QA-карточка 25) выделить управление order/visibility
     в отдельный API-контракт, чтобы тулбар мог «взаимоиспользоваться» с
     меню/гридом карточки 30: переставил столбцы прямо в таблице — элементы
     тулбара переставились (и наоборот).
   - Иконочное сокращение: если у элемента нет иконки — псевдо-иконка из
     первых 1–3 букв названия. То же для меню (компактный режим).

4. **Колонки с подколонками (группы).** Например «версии»: версия 1..4.
   Нужно уметь выбрать всю группу или отдельные подколонки. Идея: точки
   справа от иконки группы (какие «режимы» включены).

## Ограничения / принципы

- Слой самостоятельный: должен работать и с гридом, и без него
  (мобильные блоки без ag-grid вовсе).
- Не встраивать в `src/common/src/grid/agGrid4/*` — только подключение снаружи.
- Стенд: localhost:3010, проверка `tsc` + стенд (см. память о рабочем стиле).

## Исследование (интернет)

_(выполнено 2026-07-07)_

### AG Grid: API состояния колонок
- **Column State** (Community, есть и в нашей v35): `api.getColumnState()` /
  `applyColumnState(params)` / `resetColumnState()`. В `ColumnState`: `colId`,
  `hide`, `width`, `flex`, `sort`, `sortIndex`, `pinned`, rowGroup/pivot-поля.
  Фильтры НЕ входят (отдельно `getFilterModel/setFilterModel`); состояние
  групп колонок — отдельно `getColumnGroupState/setColumnGroupState`.
- **Grid State** (v31+, Community): `initialState` (читается один раз при
  создании — восстановление без «мигания»), `api.getState()`, событие
  `stateUpdated` (даёт `state` целиком + `sources`; стартовый
  `gridInitializing` игнорировать).
- v31: `columnApi` deprecated, всё слито в `gridApi` (в v35 columnApi нет).

### События для персиста
- Классика: `columnMoved`, `columnResized`, `columnVisible`, `columnPinned`,
  `sortChanged`. У `columnResized`/`columnMoved` есть **`finished: boolean`** —
  сохранять только при `finished===true` (иначе десятки событий на один drag);
  при `finished:true` поле `column` может быть `null` (известный нюанс).
- Проверять **`event.source`** — не персистить программные изменения
  (`source==='api'`), иначе цикл «restore → save».
- Debounce 100–500 мс поверх фильтра `finished`. На v31+ проще одно
  `stateUpdated` + debounce.

### Версионирование state
- Встроенного версионирования нет — уровень приложения. Паттерн: обёртка
  `{schemaVersion, savedAt, state}`.
- `applyColumnState`: неизвестные `colId` молча игнорируются, возвращает
  `false` если не все нашлись (готовый сигнал «state устарел»); колонки без
  записи в state не трогаются, если не задан `defaultState`; `undefined` =
  не трогать, `null` = сбросить; порядок — только при `applyOrder: true`.
- Миграция «мягкая»: отфильтровать записи по актуальным colId, новые колонки
  оставить дефолтными — то же, что делает `normalize()` в нашем Toolbar.

### Готовые решения
- Доминирующей npm-библиотеки нет; все решения — внешний слой над gridApi
  (события → debounce → store; restore на `gridReady`/`firstDataRendered`
  или `initialState`). Примеры: официальный блог AG Grid (React+Redux),
  репо iamtalwinder (Angular+localStorage), gist nite. Подтверждает нашу
  архитектуру «слой снаружи, обёртку не трогаем».

### Мобильный UX выбора колонок
- Конвенции индустрии: **column priority** (авто-скрытие по приоритету —
  DataTables Responsive), **column toggle/chooser** (чекбокс-панель — jQuery
  Mobile, MUI DataGrid, Material React Table), **таблица → карточки**.
- «Ползунок с точками» как паттерн выбора колонок в индустрии НЕ встречается.
  Но составляющие стандартизованы: multi-thumb slider — W3C APG паттерн
  (каждый бегунок = `role=slider`, отдельный tab-stop); точки на треке =
  discrete slider with marks (MUI); **drag-off-to-remove** — конвенция из
  редакторов градиентов (Photoshop/XD): утащил стоп с трека — удалил.
  Дискаверабилити drag-off низкая даже у дизайнеров.
- Подводные камни: touch-target ≥44×44px; близкие бегунки конфликтуют за
  палец; нужен не-жестовый эквивалент удаления; не перекрывать системный
  скролл; react-range не умеет динамически менять число бегунков (#136) —
  проще своё на pointer events.
- Card-view: заголовок карточки = главный идентификатор строки, дальше 3–5
  полей label+value, один акцент, остальное — progressive disclosure
  (разворачивание). Не выводить все колонки.

### Выводы
1. Внешний слой — правильный и общепринятый путь; на v35 доступны оба API,
   базой берём **Column State** (+опц. Grid State позже).
2. Сохранение: события с фильтром `finished` + `source!=='api'` + debounce
   ~300 мс.
3. Версионируем сами: `{schemaVersion, state}` + normalize по актуальным
   colId (паттерн уже есть в Toolbar.normalize).
4. Меню иконок (chooser) — конвенционально, делаем основным способом;
   «ползунок с точками» — как фичу-эксперимент с кнопочной альтернативой
   удаления и крупными touch-целями.
5. Карточки: видимые поля = те же выбранные колонки, 1 поле — заголовок.

### Обзор

Три слоя, каждый самостоятелен:

```
┌─ UI: ColumnsMenu (иконки) ─┐   ┌─ UI: ColumnDots + CardList (мобайл) ─┐
└────────────┬───────────────┘   └────────────────┬─────────────────────┘
             │            подписка/команды        │
        ┌────▼────────────────────────────────────▼────┐
        │  ЯДРО: createColumnState({key, columns})     │  ← персист (staticGetAdd)
        └────────────────────┬─────────────────────────┘
                             │ адаптер (подключается опционально)
                  ┌──────────▼──────────┐
                  │ attachGrid(gridApi) │  ← двусторонний sync с живым ag-grid
                  └─────────────────────┘
```

Ядро не знает про ag-grid и React (как `core.ts` у agGrid4) — мобильные
блоки работают без грида вовсе. Расположение: `src/common/src/grid/columnState/`
(рядом с agGrid4, но отдельная папка; agGrid4 не трогаем).

### Ядро: `createColumnState` (рабочее имя)

```text
type tColumnMeta = {
   key: string            // = colId в гриде
   title: string
   short?: string         // подпись для иконки/карточки
   icon?: ReactNode
   group?: string         // ключ группы (версии и т.п.)
   fixed?: boolean        // нельзя скрыть/сдвинуть (id-колонка)
   defaultVisible?: boolean
   cardRole?: 'title' | 'accent'  // роль в карточке
}

type tColumnsConfig = {
   v: number                        // schemaVersion
   order: string[]
   visible: Record<string, boolean>
   width?: Record<string, number>   // пишет только грид
   groups?: Record<string, string[]> // группа → включённые подколонки
   sort?: { key: string, dir: 'asc' | 'desc' } | null
   // sort ЛИПКИЙ: не зависит от visible/выделения; можно сортировать
   // и по скрытому полю; сбрасывается только явным переключением
   filter?: Record<string, unknown>  // filterModel грида (согласовано в v1);
   // пишет/читает только грид-адаптер (getFilterModel/setFilterModel,
   // событие filterChanged); мобильный CardList в v1 фильтры не рисует
}

createColumnState({ key, columns: tColumnMeta[], def? }) -> {
   api: {
      getConfig, setConfig, reset,
      useConfig(),                  // React-хук (подписка, как у Toolbar)
      onChange(cb) -> off,
      show(key, on), move(next: string[]),
      visibleKeys(): string[],      // в порядке order, с учётом групп
   },
   grid: {                          // адаптер, см. ниже
      attach(gridApi, opts?), detach(),
   },
}
```

- **Персист** — по конвенции репо: `staticGetAdd(key, def)` → `staticProps` →
  `Cash` → localStorage. Никаких своих localStorage-ключей.
- **normalize()** при загрузке и при смене набора `columns` (паттерн
  Toolbar.normalize): неизвестные ключи выкидываются, новые колонки
  добавляются с `defaultVisible`, `fixed` пиннятся. Это и есть миграция;
  `v` — на случай несовместимой смены формата.

### Адаптер к ag-grid (двусторонний)

- `grid.attach(gridApi)` — вызывается из пользовательского `onGridReady`
  (AgGridMy пробрасывает его поверх wiring) или через `controller.withApi`.
  При attach: `applyColumnState({state: toColumnState(config), applyOrder: true})`;
  возврат `false` → лог + normalize.
- **Грид → store**: подписка на `columnMoved`/`columnResized`/`columnVisible`/
  `sortChanged`/`filterChanged` (+`columnPinned` позже); фильтры событий:
  `finished === true`, `source !== 'api'`, debounce ~300 мс; событие фолдится
  в config (order/width/visible/sort/filter).
- **Store → грид**: `onChange` из UI → `applyColumnState`; на время применения
  ставится флаг `applying`, события с ним игнорируются (защита от цикла
  restore→save в обе стороны).
- detach — на `onGridPreDestroyed`; config переживает пересоздание грида
  (паттерн `columnBuffer.control`).

### UI 1: меню иконок `ColumnsMenu`

- Новый лёгкий компонент **по образцу `Toolbar.Settings`**, но питающийся от
  `columnState.api` (у Toolbar свой персист — переиспользовать напрямую
  нельзя, будет двойное хранение; берём его паттерны и `useReorder`).
- Рендер: иконки-колонки в порядке `order`; скрытая = приглушена/вычёркнута;
  клик = toggle видимости; drag (useReorder, holdMs для touch) = порядок.
- Группы: иконка группы + **точки справа** = включённые подколонки; клик по
  группе = вся группа, клик по точке = подколонка (фаза 8).
- Подключение: как обычный элемент или внутрь `MenuRightApi`/gear-попапа.
- Двусторонность бесплатно: меню подписано на тот же store, что и грид-адаптер.

### UI 2: мобильный режим — `ColumnDots` + `CardList`

- **`ColumnDots`** — дискретный мульти-бегунок: трек = все колонки (метки),
  точки (2–4) = выбранные. Точка стоит на метке — колонка показана.
  - Перетаскивание точки по треку = смена колонки (snap к меткам).
  - **Tap по точке (без перемещения)** = выделить поле (активная точка
    подсвечена; под треком — название поля).
  - **Кнопка «Сортировка»** рядом с треком: циклично переключает сортировку
    по выделенному полю (asc → desc → off); индикатор ↑/↓ у точки/метки
    отсортированного поля. Сортировка липкая (см. постановку 2а): смена
    выделения её не трогает, работает и по скрытому полю.
  - **Удаление** (согласовано): **свайп точки вверх** — быстрый вертикальный
    флик, точка «отрывается» с анимацией. Отличать от горизонтального drag
    по доминирующей оси первых пикселей жеста; порог скорости/дистанции,
    чтобы не конфликтовать со скроллом страницы (проверка на стенде).
  - **Добавление**: tap по пустой метке трека — точка появляется там.
  - Touch-цели ≥44px (невидимо расширенные), точки не ближе одной метки
    друг к другу.
  - Реализация своя на pointer events (react-range не умеет динамическое
    число бегунков); паттерн a11y — W3C multi-thumb slider.
  - Нюанс жестов: tap = выделение, поэтому «добавление» и «выделение»
    различаются целью tap (пустая метка vs точка) — конфликтов нет.
- **Сортировка в данных**: `CardList` сортирует по `config.sort` сам
  (comparator по типу поля); при подключённом гриде адаптер транслирует
  `sort` в column state (`sort`/`sortIndex`) и обратно.
- **`CardList`** — строки блоками: поле `cardRole:'title'` (или первая
  видимая колонка) = заголовок карточки, остальные видимые = стек
  label+value, `cardRole:'accent'` = бейдж. Данные — обычный массив/буфер,
  ag-grid не нужен.
- Оба компонента питаются от того же `columnState` → выбор на телефоне и
  состояние грида на десктопе могут жить под одним `key` или под разными
  (решить при реализации: скорее разные, `key` и `key+':mobile'`).

## Реализация: заметки (шаг 5)

- `createColumnState({key, columns, def?, saveMs?})` →
  `{api: {getConfig,setConfig,useConfig,onChange,reset,show,move,setSort,
  toggleSort,visibleKeys}, grid: {attach,detach}}`.
  ag-grid — только type-import; рантайм-связь через переданный `gridApi`.
- **Контракт загрузки**: `Cash.load()` ДО монтирования грида (attach применяет
  конфиг один раз). `addDataToMap` мержит загруженное В СУЩЕСТВУЮЩИЙ объект
  (`Object.assign`+`renderBy`), поэтому ссылка `st` из `staticGetAdd` валидна,
  но грид, приаттаченный до load, показал бы дефолты. В QA-карточке грид
  гейтится на `loaded`.
- **fixed и грид**: пользователь может перетащить колонку перед fixed прямо в
  гриде; после fromGrid-коммита normalize пиннит fixed обратно, и если порядок
  скорректирован — адаптер сразу re-apply к гриду (снап-бэк). Проверено.
- Защита от циклов: флаг `applying` + `e.source==='api'` + `finished===false`
  отсеивается + сравнение JSON до commit (skip если нет изменений).
- `autoSizeColumns={false}` у AgGridMy в карточке: авто-fit при mount/resize
  переписывал бы сохранённые ширины (источник события не 'api').
- **HMR-артефакт стенда**: hot-reload qa.tsx пересоздаёт module-level
  контроллер, а живой грид остаётся приаттачен к старому экземпляру — до F5
  store→grid не работает. В реальном приложении модуль исполняется один раз.
  Не чинится в v1; заметка для проверяющего.
